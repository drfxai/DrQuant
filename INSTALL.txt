============================================================================
 DrFX Quant - Setup & Installation Guide
============================================================================

DrFX Quant is a Telegram-style trading communication platform
(Node.js + PostgreSQL + Nginx + PM2), installed by the top-level install.sh.


============================================================================
 0. PREREQUISITES
============================================================================

  - A fresh Ubuntu/Debian server (tested on Ubuntu 24.04) with root access.
  - A domain name with an A record pointing at the server's public IP
    (required for HTTPS / Let's Encrypt).

IMPORTANT - how the code reaches your server:
  The installer runs from a `git clone` of this repository. Anything that lives
  only on a local machine reaches the server ONLY after it is committed and
  pushed to GitHub. After pushing, run `git pull` (or re-clone) on the server
  so it has the latest installer before running it.


============================================================================
 1. INSTALL (one command)
============================================================================

  git clone https://github.com/drfxai/DrFXQuant.git
  cd DrFXQuant
  sudo bash install.sh

install.sh is a one-shot installer. It will:
  - Install Node.js 20, PostgreSQL, Nginx, PM2, and (optionally) certbot.
  - Create the `drfx_quant` PostgreSQL database and a `drfx` user.
  - Copy the application to /var/www/drfx-quant (including the routes,
    public assets, middleware, services, realtime, migrations, and the
    Quantum Chat browser client served at /qc).
  - Generate /var/www/drfx-quant/.env with secure random secrets
    (JWT secret, TradingView webhook secret, etc.).
  - Create the base database tables, then apply every migration in
    migrations/*.sql (this creates the TradingView signal/webhook tables).
  - Start the app under PM2 behind an Nginx reverse proxy.
  - Optionally obtain an HTTPS certificate.
  - Install & activate the WebRTC SFU (mediasoup) + TURN (coturn) for high-FPS
    (30-60) Live Trading - ON BY DEFAULT (just press Enter at the prompt). Skip
    on a small box with:  sudo INSTALL_SFU=no bash install.sh

Prompts it will ask:
  - Domain (e.g. app.yourdomain.com)
  - Admin email
  - Admin password (min 6 chars, asked twice)
  - OpenRouter API key            (optional - press Enter to skip)
  - NowPayments API key           (optional - press Enter to skip)
  - NowPayments IPN secret        (optional - press Enter to skip)
  - Proceed? (y/n)
  - Setup SSL now? (y/n)
  - Install the Quantum Chat node now? (y/n)   (optional - see section 5)
  - Install & activate high-FPS live streaming (SFU + TURN)? [Y/n]
        (default YES - press Enter to install mediasoup + coturn for 30-60 FPS
         Live Trading; type 'n' to skip, e.g. on a 1 GB box. Preset for
         unattended installs with INSTALL_SFU=yes or INSTALL_SFU=no.)

When it finishes:
  - The app is reachable at http://your-domain  (https if you enabled SSL).
  - Log in with the admin email and password you entered.

Post-install steps for the trading features:
  - The TradingView webhook shared secret was auto-generated. Find it in:
        /var/www/drfx-quant/.env   (TRADINGVIEW_WEBHOOK_SECRET=...)
    Use that same secret in your TradingView alert payloads.
  - Create a channel named 'signals' inside the app so incoming webhook
    signals have a channel to broadcast to.
  - To enable the AI assistant, set OPENROUTER_API_KEY in the .env (re-run is
    not needed; just edit and restart). To enable crypto payments, set the
    NOWPAYMENTS_* values.

Everyday management:
  sudo bash /var/www/drfx-quant/manage.sh   # reference card: current config,
                                   # live resource usage, how to change each part
  pm2 logs drfx-quant            # view live logs
  pm2 restart drfx-quant         # restart after editing .env
  pm2 status                       # process status
  nano /var/www/drfx-quant/.env  # edit configuration
  sudo bash uninstall.sh           # remove the app


============================================================================
 2. TROUBLESHOOTING
============================================================================

App won't start / 502 Bad Gateway:
  pm2 status                       # is the process online or crash-looping?
  pm2 logs drfx-quant --lines 50 --nostream
  curl -i http://127.0.0.1:3000/   # does the backend answer locally?
  - "Cannot find module './realtime/...'" -> the realtime/ directory is missing
    from /var/www/drfx-quant. Re-run install.sh (it copies realtime/ and now
    fails fast if it is absent), or copy the folder in from a fresh clone.
  - "ECONNREFUSED 127.0.0.1:5432" / DB auth errors -> PostgreSQL isn't running
    or DATABASE_URL in .env is wrong.

TradingView webhook returns 500:
  - Ensure migrations were applied (install.sh "Step 3b"); the signals and
    webhook_logs tables must exist.

Login / admin not working:
  - The admin user is created/updated from ADMIN_EMAIL / ADMIN_PASSWORD in
    /var/www/drfx-quant/.env on each boot.

PostgreSQL:
  sudo -u postgres psql -d drfx_quant -c "\dt"     # list tables


============================================================================
 3. UPDATING (in place, no full reinstall)
============================================================================

  cd /path/to/DrFXQuant
  git pull                    # fetch the new version (update.sh also tries this)
  sudo bash update.sh

  update.sh refreshes an existing install WITHOUT re-running first-time setup:
    - Preserves your .env (domain, admin, secrets), the database + all data,
      and the uploads/ folder.
    - Refreshes the application code and Node dependencies.
    - Applies only NEW database migrations (tracked in a schema_migrations
      table; already-applied ones are skipped). On a migration error it stops
      BEFORE restarting, so the currently-running version keeps serving.
    - Backs up .env first, then restarts the app under PM2.
    - Prints the management reference card at the end.

  Re-running install.sh also works and is idempotent, but it re-prompts for
  first-time configuration; update.sh is the quicker path for upgrades.


============================================================================
 4. FILE / PATH REFERENCE
============================================================================

  Install dir      /var/www/drfx-quant
  Config           /var/www/drfx-quant/.env            (chmod 600)
  Process manager  PM2 process name: drfx-quant
  Web server       Nginx, proxying to 127.0.0.1:3000
  Database         PostgreSQL db "drfx_quant", user "drfx"

============================================================================
 5. QUANTUM CHAT (optional)
============================================================================

Quantum Chat is a DNS-resilient, end-to-end-encrypted "emergency" messenger.
It is a SEPARATE service from the main platform and is designed to keep working
even if the main site is blocked (and vice-versa). Keys, contacts, and messages
stay in the browser; they are never sent to the Node backend or the Postgres DB.

Two parts:

  a) The in-app panel (always available).
     A "Quantum Chat" item appears in the in-app menu (sidebar). It opens the
     browser client (served by the main app at /quantum-chat.html, using the
     module at /qc/quantum-chat.js). install.sh deploys this automatically.
     The panel lets a user generate an ID, add/verify contacts by safety
     number, and exchange messages - but it needs a running node to connect to.

  b) The node (optional, installed separately).
     install.sh "Step 7" offers to install the Quantum Chat node. You can also
     run it later:
        sudo bash quantum-chat/scripts/install-quantum-chat.sh

     The node is a DNS service. It requires, OUTSIDE any installer:
       - A subdomain delegated to this server (e.g. an NS record so that
         qc.yourdomain.com is served by ns1.yourdomain.com, with an A record
         for ns1 pointing at the server's public IP).
       - Your hosting provider's firewall to allow UDP and TCP port 53.
     The installer can free a local port 53 held by systemd-resolved, but it
     cannot create DNS delegation or open provider-level firewalls for you.

Honest status: the browser client is reference code that mirrors the verified
Go node byte-for-byte but has not been browser-tested against a live node here;
identities created in the panel currently last for the session only. Treat
Emergency Mode as best-effort (it works when DNS-over-HTTPS is reachable, not
when all HTTPS is blocked), and always compare a contact's safety number out of
band before trusting it.
